在服务器终端执行以下命令：

# 1. 更新系统
sudo apt update && sudo apt upgrade -y

# 2. 安装基础软件

# 3. 安装 MySQL 客户端（用于测试数据库连接）
sudo apt install -y mysql-client

# 4. 安装 Node.js 18+（用于构建前端，如果选择在服务器构建）
# 
# 方法1：使用 NodeSource 仓库（推荐，如果网络正常）
# 注意：这两个命令必须分开执行，或者用 && 连接
# curl -fsSL https://deb.nodesource.com/setup_18.x | sudo bash -
# sudo apt install -y nodejs
#
# 如果方法1卡住或失败，使用以下替代方案：

# 方法2：使用 Ubuntu 官方源（版本可能较旧，但稳定）
sudo apt update
sudo apt install -y nodejs npm

# 方法3：使用 NVM（Node Version Manager，推荐用于开发环境）
# curl -o- https://raw.githubusercontent.com/nvm-sh/nvm/v0.39.0/install.sh | bash
# source ~/.bashrc
# nvm install 18
# nvm use 18

# 方法4：手动下载安装（如果网络有问题）
# wget https://nodejs.org/dist/v18.20.0/node-v18.20.0-linux-x64.tar.xz
# tar -xf node-v18.20.0-linux-x64.tar.xz
# sudo mv node-v18.20.0-linux-x64 /opt/nodejs
# sudo ln -s /opt/nodejs/bin/node /usr/local/bin/node
# sudo ln -s /opt/nodejs/bin/npm /usr/local/bin/npm

# 5. 验证安装
python3 --version  # 应该显示 Python 3.10+
node --version     # 应该显示 v18.x 或更高
nginx -v           # 应该显示 nginx 版本

**注意事项**：

- **锁文件错误**：如果遇到 `Could not get lock /var/lib/dpkg/lock-frontend`，这是正常的，表示另一个 apt 进程正在运行。等待 1-2 分钟后重试即可。

- **内核升级提示**：在安装软件包时，系统可能会弹出"Pending kernel upgrade"（待处理的内核升级）对话框。这表示：
  - 系统已安装了新版本的内核（例如从 5.15.0-160-generic 升级到 5.15.0-164-generic）
  - 当前运行的是旧内核，需要重启服务器才能使用新内核
  - **这是正常现象**，不影响当前安装过程
  - 点击 `<Ok>` 关闭对话框即可继续安装
  - **建议**：完成所有软件安装和配置后，再重启服务器以加载新内核

## 配置后端为系统服务
ash
# 创建 systemd 服务文件
sudo tee /etc/systemd/system/procurement-api.service > /dev/null << 'EOF'
[Unit]
Description=Procurement Manager API
After=network.target

[Service]
Type=notify
User=ubuntu
Group=ubuntu
WorkingDirectory=/var/www/procurement-manager/backend
Environment="PATH=/var/www/procurement-manager/backend/venv/bin"
ExecStart=/var/www/procurement-manager/backend/venv/bin/gunicorn \
    -w 4 \
    -k uvicorn.workers.UvicornWorker \
    --bind 127.0.0.1:8000 \
    --timeout 120 \
    --access-logfile - \
    --error-logfile - \
    app.main:app
Restart=always
RestartSec=3

[Install]
WantedBy=multi-user.target
EOF

# 重新加载 systemd
sudo systemctl daemon-reload

# 启动服务
sudo systemctl enable procurement-api
sudo systemctl start procurement-api

# 查看服务状态
sudo systemctl status procurement-api

# 查看日志（实时）
sudo journalctl -u procurement-api -f
# 按 Ctrl+C 退出日志查看---

## 配置 Nginx

# 创建 Nginx 配置文件
sudo tee /etc/nginx/sites-available/procurement-manager > /dev/null << 'EOF'
server {
    listen 80;
    server_name gaohuaiyu.vip www.gaohuaiyu.vip;

    # 前端静态文件目录
    root /var/www/procurement-manager/frontend/dist;
    index index.html;

    # 前端路由支持（React Router）
    location / {
        try_files $uri $uri/ /index.html;
    }

    # 后端API代理
    location /api/ {
        proxy_pass http://127.0.0.1:8000;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;
        
        # 文件上传大小限制
        client_max_body_size 10M;
        
        # 超时设置
        proxy_connect_timeout 60s;
        proxy_send_timeout 60s;
        proxy_read_timeout 60s;
    }

    # 静态资源缓存
    location ~* \.(js|css|png|jpg|jpeg|gif|ico|svg|woff|woff2|ttf|eot)$ {
        expires 1y;
        add_header Cache-Control "public, immutable";
    }

    # 日志
    access_log /var/log/nginx/procurement-access.log;
    error_log /var/log/nginx/procurement-error.log;
}
EOF

# 启用配置
sudo ln -s /etc/nginx/sites-available/procurement-manager /etc/nginx/sites-enabled/

# 删除默认配置（可选）
sudo rm /etc/nginx/sites-enabled/default

# 测试配置
sudo nginx -t

# 如果测试通过，重启 Nginx
sudo systemctl restart nginx

# 查看 Nginx 状态
sudo systemctl status nginx---

#### 5.1 apt 锁文件错误

如果执行 `apt update` 时遇到：
```
E: Could not get lock /var/lib/dpkg/lock-frontend
```

**原因**：另一个 apt 进程正在运行。

**解决方案**：
```bash
# 等待 1-2 分钟后重试
sudo apt update

# 或者检查是否有其他 apt 进程
ps aux | grep apt

# 如果有残留进程，可以结束它（谨慎操作）
sudo killall apt apt-get
sudo dpkg --configure -a
```

#### 5.2 dpkg 被中断错误

如果执行 `apt update` 时遇到：
```
E: dpkg was interrupted, you must manually run 'sudo dpkg --configure -a' to correct the problem.
```

**原因**：之前的包安装或更新过程被意外中断（如网络断开、系统重启、手动终止等），导致 dpkg 状态不一致。

**解决方案**：

**步骤1：修复 dpkg 状态**
```bash
sudo dpkg --configure -a
```
这个命令会完成之前被中断的包配置过程。

**步骤2：如果步骤1完成后仍有问题，清理并修复**
```bash
# 清理包缓存
sudo apt-get clean

# 修复依赖关系
sudo apt-get install -f

# 更新包列表
sudo apt-get update
```

**步骤3：重新执行更新**
```bash
sudo apt update && sudo apt upgrade -y
```

**如果上述方法仍无法解决**，可以尝试：
```bash
# 删除锁文件（谨慎操作，确保没有其他 apt 进程在运行）
sudo rm /var/lib/dpkg/lock
sudo rm /var/lib/dpkg/lock-frontend
sudo rm /var/cache/apt/archives/lock

# 然后重新配置
sudo dpkg --configure -a
sudo apt-get update
```

**预防措施**：
- 避免在包管理过程中强制终止进程
- 确保网络连接稳定
- 避免在系统更新时重启服务器

### 问题6：sudo 无法执行错误

如果执行命令时遇到：
```
/usr/bin/sudo: /usr/bin/sudo: cannot execute binary file
```

**原因**：sudo 二进制文件可能损坏、架构不匹配、文件系统挂载为 noexec，或 sudo 包安装不完整。

**解决方案**：

**步骤1：检查 sudo 文件状态**
```bash
# 检查 sudo 文件是否存在和权限
ls -l /usr/bin/sudo

# 检查文件类型和架构
file /usr/bin/sudo

# 检查系统架构
uname -m
```

**步骤2：如果当前是 root 用户，直接修复 sudo**
```bash
# 检查是否已经是 root（提示符应该是 # 而不是 $）
whoami

# 如果是 root，重新安装 sudo
apt-get update
apt-get install --reinstall sudo

# 修复权限
chmod 4755 /usr/bin/sudo
```

**步骤3：如果不是 root 用户，需要切换到 root**
```bash
# 尝试使用 su 切换到 root（需要 root 密码）
su -

# 或者如果配置了 root 登录，直接使用 root 账户登录
# 然后执行步骤2的命令
```

**步骤4：检查文件系统挂载选项**
```bash
# 检查 /usr 目录是否挂载为 noexec
mount | grep /usr

# 如果是 noexec，需要重新挂载（需要 root 权限）
mount -o remount,exec /usr
```

**步骤5：如果 sudo 完全损坏，使用 root 账户**
```bash
# 如果无法修复 sudo，可以：
# 1. 使用 root 账户直接登录
# 2. 或者使用 su 切换到 root
# 3. 然后重新安装 sudo
apt-get update
apt-get install --reinstall sudo
apt-get install --reinstall sudo-ldap  # 如果使用了 LDAP
```

**步骤6：修复后验证**
```bash
# 测试 sudo 是否正常工作
sudo -v

# 如果成功，继续安装 Node.js
# 注意：这两个命令必须分开执行，不要连在一起
curl -fsSL https://deb.nodesource.com/setup_18.x | sudo bash -
sudo apt install -y nodejs
```

**常见错误**：如果遇到 `/usr/bin/sudo: cannot execute binary file` 错误，可能是因为命令格式错误：
- ❌ 错误：`curl ... | sudo bash - sudo apt install -y nodejs`（两个命令连在一起）
- ✅ 正确：`curl ... | sudo bash -` 然后 `sudo apt install -y nodejs`（分开执行）
- ✅ 正确：`curl ... | sudo bash - && sudo apt install -y nodejs`（用 && 连接）

**替代方案：不使用 sudo（如果当前是 root 用户）**

如果当前已经是 root 用户，可以省略 sudo：
```bash
# 直接执行，不需要 sudo
curl -fsSL https://deb.nodesource.com/setup_18.x | bash -
apt install -y nodejs
```

**预防措施**：
- 避免强制终止包管理进程
- 定期更新系统以保持 sudo 包完整
- 避免手动修改 /usr/bin/sudo 文件

### 问题7：Node.js 安装脚本卡住无响应

如果执行 `curl -fsSL https://deb.nodesource.com/setup_18.x | sudo bash -` 时命令卡住，没有任何输出：

**原因**：
- 网络连接问题，无法访问 deb.nodesource.com
- DNS 解析失败
- 防火墙/代理阻止连接
- 脚本执行时等待用户输入但被阻塞

**解决方案**：

**步骤1：诊断网络连接**
```bash
# 测试网络连接
ping -c 3 deb.nodesource.com

# 测试 DNS 解析
nslookup deb.nodesource.com

# 测试 HTTPS 连接
curl -I https://deb.nodesource.com/setup_18.x

# 如果上述命令都失败，可能是网络问题
```

**步骤2：如果网络有问题，使用替代安装方法**

**方法A：使用 Ubuntu 官方源（最简单）**
```bash
sudo apt update
sudo apt install -y nodejs npm

# 验证安装
node --version
npm --version

# 注意：Ubuntu 官方源的 Node.js 版本可能较旧（如 12.x 或 14.x）
# 如果版本太低，需要使用其他方法
```

**方法B：使用 NVM（Node Version Manager，推荐）**
```bash
# 下载并安装 NVM
curl -o- https://raw.githubusercontent.com/nvm-sh/nvm/v0.39.0/install.sh | bash

# 重新加载 shell 配置
source ~/.bashrc

# 安装 Node.js 18
nvm install 18
nvm use 18

# 设置为默认版本
nvm alias default 18

# 验证安装
node --version
npm --version
```

**方法C：手动下载安装（如果网络受限）**
```bash
# 下载 Node.js 二进制包（替换为最新版本号）
cd /tmp
wget https://nodejs.org/dist/v18.20.0/node-v18.20.0-linux-x64.tar.xz
```

**解决方案C：检查防火墙配置（重要说明）**

**关键理解**：
- **入站流量（Inbound）**：从外部进入服务器的流量（由防火墙规则控制）
- **出站流量（Outbound）**：从服务器向外发出的流量（默认允许，除25端口）

**git clone 的连接方向**：
- `git clone git@github.com` 是从服务器**向外连接**GitHub，属于**出站流量**
- 云服务器防火墙主要控制**入站流量**，对出站流量的限制较少

**检查云服务器防火墙配置**：
1. 登录云服务器控制台
2. 查看防火墙规则：
   - 端口22（SSH）：通常已允许入站（用于服务器登录）
   - 端口443（HTTPS）：通常已允许入站（用于HTTPS服务）
   - **但这些规则不影响出站连接**

**如果防火墙显示端口22和443都已允许，但 git clone 仍然卡住**：
- 说明问题不在服务器防火墙
- 可能是云服务商在**网络层面**限制了出站SSH连接（端口22）
- 或者GitHub的SSH服务（端口22）在某些网络环境下不可达

**解决方案**：
- 使用 HTTPS 方式克隆（端口443，通常不会被限制）
- 或配置 SSH 使用 HTTPS 端口（443）连接

**推荐方案**：
- **SSH 端口被阻止**：使用解决方案A（HTTPS方式），最简单可靠
- **如果网络正常**：使用方法E（浅克隆）最快
- **如果网络有问题**：使用方法A（SCP 上传）或方法B（压缩包）
- **必须使用 SSH**：使用解决方案B（配置 SSH 使用 443 端口）

### 问题11：SSH 连接经常断开（FinalShell 或其他 SSH 客户端）

如果使用 FinalShell、PuTTY 或其他 SSH 客户端连接服务器时经常断开连接：

**原因**：
- SSH 服务器端默认会在空闲一段时间后断开连接
- TCP keepalive 未启用
- 网络不稳定
- 客户端超时设置过短

**解决方案**：

**步骤1：配置服务器端 SSH（防止服务器主动断开）**

在服务器上执行：
```bash
# 1. 备份原配置文件
sudo cp /etc/ssh/sshd_config /etc/ssh/sshd_config.backup

# 2. 编辑 SSH 配置文件
sudo nano /etc/ssh/sshd_config

# 3. 找到并修改以下配置（如果不存在则添加）：
# ClientAliveInterval 60        # 每60秒发送一次心跳
# ClientAliveCountMax 3         # 最多发送3次，总共3分钟无响应才断开
# TCPKeepAlive yes              # 启用 TCP keepalive

# 或者设置为更长的超时时间：
ClientAliveInterval 300        # 每5分钟发送一次心跳
ClientAliveCountMax 2          # 最多发送2次，总共10分钟无响应才断开
TCPKeepAlive yes

# 4. 保存文件（Ctrl+O, Enter, Ctrl+X）

# 5. 重启 SSH 服务使配置生效
sudo systemctl restart sshd

# 6. 验证配置
sudo sshd -t  # 检查配置语法是否正确
```

**步骤2：配置 FinalShell 客户端**

1. **打开 FinalShell 设置**：
   - 点击 FinalShell 菜单栏的"选项" → "选项"
   - 或右键服务器连接 → "编辑" → "高级"

2. **配置连接保持**：
   - **保持连接**：勾选"启用保持连接"
   - **发送心跳间隔**：设置为 30-60 秒（例如 30）
   - **超时时间**：设置为 300-600 秒（例如 300）

3. **SSH 高级设置**：
   - 在"SSH"选项卡中：
     - **连接超时**：设置为 60 秒或更长
     - **保持连接**：启用
     - **自动重连**：启用（可选）

**步骤3：配置客户端 SSH 配置文件（可选，适用于所有 SSH 客户端）**

在本地电脑创建或编辑 SSH 配置文件：

**Windows（FinalShell 或其他客户端）**：
```bash
# 如果使用 FinalShell，配置文件通常在：
# C:\Users\你的用户名\.ssh\config

# 或者创建配置文件：
# C:\Users\你的用户名\.ssh\config
```

**Linux/Mac**：
```bash
# 编辑或创建 ~/.ssh/config
nano ~/.ssh/config
```

添加以下内容：
```
Host *
    ServerAliveInterval 60      # 每60秒发送一次心跳
    ServerAliveCountMax 3       # 最多发送3次
    TCPKeepAlive yes            # 启用 TCP keepalive
    ConnectTimeout 30           # 连接超时30秒
```

**针对特定服务器**：
```
Host 114.132.198.141
    HostName 114.132.198.141
    User ubuntu
    ServerAliveInterval 60
    ServerAliveCountMax 3
    TCPKeepAlive yes
    ConnectTimeout 30
```

```bash
# 如果延迟不稳定或丢包，可能是网络问题
# 考虑使用更稳定的网络连接
```

**推荐配置**：

**服务器端（/etc/ssh/sshd_config）**：
```
ClientAliveInterval 300        # 5分钟心跳
ClientAliveCountMax 2          # 10分钟无响应才断开
TCPKeepAlive yes
```

**客户端（FinalShell）**：
- 保持连接：启用
- 发送心跳间隔：30 秒
- 超时时间：300 秒

**验证配置**：
```bash
# 1. 检查 SSH 服务状态
sudo systemctl status sshd

# 2. 测试连接稳定性
# 连接后，等待一段时间不操作，看是否还会断开

# 3. 使用 screen/tmux 测试
screen -S test
# 等待10分钟不操作，然后断开连接
# 重新连接后执行 screen -r test，应该能恢复会话
```

**如果仍然频繁断开**：
1. 检查云服务商是否有网络限制
2. 检查防火墙规则
3. 考虑使用 VPN 或更稳定的网络
4. 使用 screen/tmux 作为备用方案
